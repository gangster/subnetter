# Subnetter Architecture

This document describes the technical architecture of the Subnetter tool, including its components, data flow, and design decisions.

## System Overview

Subnetter is a tool for allocating IPv4 CIDR blocks across cloud infrastructure in a hierarchical, deterministic manner. It takes a configuration file as input and produces a set of non-overlapping CIDR allocations for subnets across multiple cloud providers, accounts, regions, and availability zones.

The tool is designed to be:

1. **Deterministic**: The same input configuration always produces the same output allocations
2. **Hierarchical**: CIDRs are allocated in a nested structure (accounts → regions → AZs → subnets)
3. **Non-overlapping**: All allocated CIDRs are guaranteed not to overlap
4. **Cloud-agnostic**: Works with AWS, Azure, GCP, and other cloud providers
5. **Flexible**: Supports various configuration options and output formats

## Flow from Configuration File to Output

The high-level flow of the tool is:

1. **Load Configuration**: Parse and validate the JSON/YAML configuration file
2. **Normalize Configuration**: Convert the configuration to a standardized internal format
3. **Allocate CIDRs**: Recursively allocate CIDR blocks at each level of the hierarchy
4. **Generate Output**: Format the allocations as CSV or other output formats

## Cloud Provider Detection

Although Subnetter is designed to be cloud-agnostic, it includes specialized logic to handle differences between major cloud providers (AWS, Azure, GCP). The provider detection system allows for region-specific optimizations and provider-specific naming conventions.

Key components:

1. **RegionDetector**: A utility that identifies the cloud provider based on region name patterns
   - AWS regions typically follow `us-east-1`, `eu-west-2` patterns
   - Azure regions follow `eastus`, `westeurope` patterns
   - GCP regions follow `us-central1`, `europe-west1` patterns

2. **AzHelper**: Provider-specific logic for generating availability zone names
   - For AWS: Appends letters to region names (e.g., `us-east-1a`, `us-east-1b`)
   - For Azure: Uses numbered availability zones (e.g., `1`, `2`, `3`)
   - For GCP: Uses zone letters (e.g., `us-central1-a`, `us-central1-b`)

3. **Default Fallback**: If a region doesn't match known patterns, AWS is used as the default provider

The cloud provider detection system is implemented in:
- `packages/core/src/utils/region-detector.ts`: Pattern matching logic
- `packages/core/src/allocator/utils/cloud/provider.ts`: Provider detector class
- `packages/core/src/allocator/utils/cloud/az.ts`: AZ naming based on provider

This detection mechanism enables the allocator to generate appropriate availability zone names and apply provider-specific networking constraints without requiring explicit provider configuration.

Key validation rules:

1. **CIDR Format**:
   - Base CIDR must be a valid IPv4 CIDR block (e.g., `10.0.0.0/8`)
   - Custom account CIDRs must also be valid IPv4 CIDR blocks
   - CIDR validation uses regex: `/^([0-9]{1,3}\.){3}[0-9]{1,3}\/([0-9]|[1-2][0-9]|3[0-2])$/`

2. **Prefix Length Constraints**:
   - Account prefix must be between 8 and 28
   - Region prefix must be between 16 and 28
   - AZ prefix must be between 19 and 28
   - Subnet prefix must be between 20 and 29
   - Hierarchical constraint: `account < region < az < subnet`

3. **Account Validation**:
   - Account names must not be empty
   - Each account must have at least one region defined
   - Account names must be unique

4. **Subnet Types**:
   - Must be defined as an object with type names as keys and prefix lengths as values
   - Each subnet type must have a unique name
   - Prefix length must be a number between 20 and 29

5. **Space Allocation**:
   - The system validates that there is enough address space for the requested allocation
   - For example, if you request 5 accounts with a /16 prefix, your base CIDR must be at least a /11

These validation rules help prevent configuration errors that would lead to impossible or invalid allocations, ensuring that the CIDR allocator can successfully generate non-overlapping network blocks.

### Logging System Features

1. **Log Levels**: The logger supports five levels of verbosity:
   - `ERROR`: Critical issues that prevent the tool from functioning
   - `WARN`: Potential issues that don't prevent operation but might need attention
   - `INFO`: Standard operational information (default level)
   - `DEBUG`: Detailed information for troubleshooting
   - `TRACE`: Very detailed trace information for deep debugging

2. **Contextual Logging**: Each log entry includes:
   - Timestamp
   - Log level
   - Source component
   - Message
   - Optional additional context data

3. **Source Attribution**: Loggers are created with a specific source name, allowing messages to be traced to their originating component:

```typescript
const logger = createLogger('CidrAllocator');
logger.info('Starting allocation process');
```

4. **Configuration**: The logging system can be configured through:
   - Environment variables (`LOG_LEVEL`)
   - Programmatic configuration via `configureLogger()`
   - CLI arguments (`--log-level`)

The logging system is implemented in `packages/core/src/utils/logger.ts` and is used throughout the codebase to provide visibility into the allocation process.

### Key Observability Points

Key points in the system that are logged include:

1. **Configuration Loading**: Log information about the loaded configuration file
2. **Validation Results**: Log validation errors and warnings
3. **Allocation Steps**: Log the progress of the allocation process
4. **Output Generation**: Log information about the generated output

Observability best practices when using Subnetter:

1. Use `--log-level debug` for more detailed information during troubleshooting
2. Check warnings, as they may indicate potential issues even if the allocation succeeds
3. For large configurations, consider piping output to a log file: `subnetter generate --log-level debug > allocation.log`

### Time Complexity

The time complexity of the CIDR allocation algorithm is linear with respect to the number of allocations:

`O(n × r × a × s)` where:
- `n` = number of accounts
- `r` = average number of regions per account
- `a` = average number of availability zones per region
- `s` = average number of subnet types

The allocation process follows a hierarchical pattern:
1. For each account: `O(n)`
2. For each region in the account: `O(r)`
3. For each AZ in the region: `O(a)`
4. For each subnet type in the AZ: `O(s)`

### Key Performance Considerations

1. **CIDR Subdivision Operations**: Each level of the hierarchy involves CIDR subdivision operations that are constant time `O(1)` but involve complex subnet calculations.

2. **Validation Checks**: Throughout the allocation process, validation checks are performed to ensure CIDRs don't overlap, which adds a small overhead.

3. **Memory Usage**: For large configurations, memory usage scales linearly with the number of allocations.

4. **Performance Bottlenecks**:
   - Very large configurations (hundreds of accounts or thousands of subnets)
   - Complex CIDR calculations for unusual prefix length combinations
   - Configuration validation for deeply nested structures

### Benchmarks

Typical performance characteristics on modern hardware:

| Configuration Size | Accounts | Regions | AZs | Subnet Types | Approximate Time |
|-------------------|----------|---------|-----|--------------|------------------|
| Small             | 1-5      | 2-3     | 2-3 | 3-5          | < 100ms          |
| Medium            | 5-20     | 3-10    | 3   | 4-8          | 100-500ms        |
| Large             | 20-100   | 5-20    | 3   | 5-10         | 0.5-2s           |
| Very Large        | 100+     | 20+     | 3+  | 10+          | 2s+              |

### Optimization Strategies

1. **Efficient Schema Validation**: The schema validation is optimized to fail fast on invalid configurations.

2. **Caching**: The system caches intermediate CIDR calculations to avoid redundant computations.

3. **Single-pass Allocation**: The allocation algorithm is designed to make a single pass through the configuration hierarchy.

4. **Tracking Optimizations**: The CIDR tracking system uses efficient data structures to minimize lookup times.

### Common Error Types and Recovery Strategies

1. **Configuration Validation Errors**
   - **Issue**: Schema validation failures
   - **Recovery**:
     - Detailed error messages point to the exact field with the issue
     - Zod validation provides expected type/format information
     - Run with `--validate-only` flag to check configuration without attempting allocation

2. **CIDR Calculation Errors**
   - **Issue**: Invalid CIDR operations (e.g., subnet doesn't fit in parent)
   - **Recovery**:
     - Check prefix lengths in hierarchy
     - Ensure base CIDR is large enough for allocation
     - Use larger CIDR blocks or reduce the number of allocations

3. **Allocation Errors**
   - **Issue**: Not enough space for requested allocations
   - **Recovery**:
     - Increase base CIDR size
     - Adjust prefix lengths to use less space
     - Reduce number of accounts/regions/AZs

4. **Output Errors**
   - **Issue**: Problems writing output files
   - **Recovery**:
     - Check permissions for output directory
     - Verify disk space
     - Use stdout output (`-o -`) instead of file

### Built-in Error Handling Features

1. **Contextual Errors**: All errors include context about where they occurred
   ```
   Error allocating CIDR for region 'us-east-1' in account 'prod':
   Not enough space in parent CIDR '10.0.0.0/16' for 3 subnets with prefix length 24
   ```

2. **Recovery Suggestions**: Many errors include actionable suggestions
   ```
   Consider using a larger base CIDR (e.g., /15 instead of /16)
   or reducing the number of subnets/regions
   ```

3. **Validation-Only Mode**: Run with `--validate-only` to check configuration without allocation

4. **Verbose Mode**: Use `--log-level debug` to get detailed information about each step

### Example Recovery Scenarios

1. **Not Enough Space**:
   ```
   Error: Not enough space in parent CIDR '10.0.0.0/8' for 256 accounts with prefix length /16
   ```
   Recovery: Either reduce the number of accounts or use a smaller prefix length for accounts (e.g., /18)

2. **Invalid CIDR Format**:
   ```
   Error: Invalid CIDR format: '10.0.0/8' at config.baseCidr
   ```
   Recovery: Fix the CIDR format to include all octets: '10.0.0.0/8'

3. **Overlapping CIDRs**:
   ```
   Error: Account 'prod' CIDR '10.0.0.0/16' overlaps with account 'dev' CIDR '10.0.0.0/17'
   ```
   Recovery: Use non-overlapping CIDRs for custom account CIDR assignments

### Backward Compatibility Principles

1. **Additive Schema Changes**: New fields are added in a way that doesn't break existing configurations
   - New fields include default values or are optional
   - Required fields are not added in minor versions

2. **Schema Versioning**: Major schema changes increment the major version
   - Version 1.x.x: Basic schema with account-region-AZ hierarchy
   - Version 2.x.x: Added multi-cloud support
   - Future versions will follow semantic versioning

3. **Format Normalization**: Config formats are normalized internally
   - Example: The subnet types format changed from array to object format, but both are supported
   - Normalizers convert legacy formats to current internal representations

4. **Deprecation Cycle**: Changes follow a deprecation cycle:
   1. New format introduced with backward compatibility
   2. Warning logs for deprecated format usage
   3. Removal of support for deprecated format in a future major version

### Example Schema Evolution

Here's an example of how the subnet types schema evolved:

**Original Format (Array):**

**Current Format (Object):**
```json
"subnetTypes": {
  "Public": 26,
  "Private": 27,
  "Data": 28
```

The system supports both formats through a normalization function that converts the array format to the object format internally before processing.

### Migration Utilities

1. **Validation Mode**: Use `--validate-only` to check if a configuration is valid
2. **Verbose Mode**: Use `--log-level debug` to get warnings about deprecated formats
3. **Migration Examples**: Documentation includes examples of migrating between formats

### Future Evolution Strategy

As the schema continues to evolve, Subnetter will maintain these principles:

1. **Document All Changes**: Clear documentation for each schema change
2. **Provide Migration Paths**: Tools and examples for migrating configurations
3. **Follow Semantic Versioning**: Breaking changes only in major versions
4. **Automatic Format Detection**: Support for multiple formats where possible
