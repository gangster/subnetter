Architecture

Subnetter Architecture

This document describes the technical architecture of the Subnetter tool, including its components, data flow, and design decisions.

System Overview

Subnetter is a tool for allocating IPv4 CIDR blocks across cloud infrastructure in a hierarchical, deterministic manner. It takes a configuration file as input and produces a set of non-overlapping CIDR allocations for subnets across multiple cloud providers, accounts, regions, and availability zones.

The tool is designed to be:

  1. Deterministic: The same input configuration always produces the same output allocations
  2. Hierarchical: CIDRs are allocated in a nested structure (accounts → regions → AZs → subnets)
  3. Non-overlapping: All allocated CIDRs are guaranteed not to overlap
  4. Cloud-agnostic: Works with AWS, Azure, GCP, and other cloud providers
  5. Flexible: Supports various configuration options and output formats

Flow from Configuration File to Output

The high-level flow of the tool is:

  1. Load Configuration: Parse and validate the JSON/YAML configuration file
  2. Normalize Configuration: Convert the configuration to a standardized internal format
  3. Allocate CIDRs: Recursively allocate CIDR blocks at each level of the hierarchy
  4. Generate Output: Format the allocations as CSV or other output formats

Core Components

Configuration Validator

The configuration validator is responsible for:

  1. Loading the configuration file (JSON or YAML)
  2. Validating the structure and values against a schema
  3. Normalizing the configuration for internal use

Key files:

  • packages/core/src/config/loader.ts: Loads and validates configuration files
  • packages/core/src/config/schema.ts: Defines the configuration schema using Zod

CIDR Allocator

The CIDR allocator is the heart of the system, responsible for:

  1. Dividing the base CIDR block into account-level blocks
  2. Further subdividing into regions, AZs, and subnets
  3. Tracking allocated CIDRs to prevent overlaps
  4. Generating the final allocation objects

Data Model

The core data model consists of these key interfaces:

classDiagram class Config { +string baseCidr +PrefixLengths prefixLengths +string[] cloudProviders +Account[] accounts +SubnetTypesMap subnetTypes } class PrefixLengths { +number account +number region +number az } class Account { +string name +Record clouds } class CloudConfig { +string baseCidr +string[] regions } class SubnetTypesMap { +Record~string, number~ map } class Allocation { +string accountName +string vpcName +string cloudProvider +string regionName +string availabilityZone +string regionCidr +string vpcCidr +string azCidr +string subnetCidr +string cidr +string subnetRole +number usableIps } Config "1" --> "*" Account Config "1" --> "1" SubnetTypesMap Config "1" --> "0..1" PrefixLengths Account "1" --> "*" CloudConfig

These interfaces define the structure of the configuration input and allocation output. The subnetTypes field in the Config interface is a simple mapping of subnet type names to prefix lengths.

Error Handling

The system uses a hierarchical error handling approach:

classDiagram class Error { +string message +string name } class ConfigValidationError { +string message +ZodError errors } class CidrError { +string message } class AllocationError { +string message } class OutputError { +string message } Error <|-- ConfigValidationError Error <|-- CidrError Error <|-- AllocationError Error <|-- OutputError
  1. Domain-specific errors: CidrError, AllocationError, ConfigValidationError
  2. Contextual error wrapping: Errors are wrapped with context about where they occurred
  3. Helpful error messages: Error messages include specific details about what went wrong
